# Beecrasy 蜜蜂基因组系统 —— 玩法与设计文档

---

## 一、设计愿景

Beecrasy 的基因组系统是模组蜜蜂养殖玩法的核心骨架。它借用现实遗传学的概念——基因、等位基因、二倍体、突变、自然选择——但并不追求严格的生物学仿真，而是在 **可玩性、可扩展性和直观性** 之间取得平衡。

玩家通过**选择性育种**和**环境调控**来影响这些性状，逐步培育出适应特定环境、产出特定物品的蜜蜂品系。

---

## 二、基因与性状

蜜蜂基因组包含七个基因位点，每个位点控制一个性状。以下按玩家实际感知的重要性逐一介绍。

### 2.1 产品序列（Products）

**这是整个系统最重要的基因。** 产品序列是一个**有序列表**，记录了蜜蜂能产出的所有物品。列表中的每项产品包含三个要素：

- **关联的生境**：该产品在哪种生境下才能产出；
- **物品模板**：产出的具体物品及其数量；
- **配方来源**（可选）：如果该产品是通过合成或熔炼突变产生的，会记录原始配方 ID。

产品序列的独特之处在于，它承载了"进化史"——一只蜜蜂的产品序列越长，说明它经历的杂交和突变越多。但序列长度有上限（9 项），且列表的前后顺序可能影响合成突变的匹配。

野生蜜蜂通常只有一项产品（如蜂蜜、蜜脾），而经过玩家多代培育的蜜蜂可能产出矿石、作物甚至稀有材料。

### 2.2 生境（Biotope）

生境决定了蜜蜂需要在什么类型的环境方块附近才能正常工作。游戏内置四种生境：

| 生境 |
|------|
| 野生（wild） |
| 合成（craft） |
| 熔炼（smelt） |
| 共生（creature） |

生境方块由方块标签（如 `flowers/biotope_wild`）定义，模组包作者可以自由配置。蜜蜂工作时，系统会在蜂箱周围扫描对应生境的方块。

生境还与**突变**深度绑定——合成生境的蜜蜂可能触发合成突变，将产品序列中的物品在工作台上合成；熔炼生境的蜜蜂可能触发熔炼突变。

### 2.3 温度（Temperature）

温度基因决定蜜蜂能适应的气候温度区间。内置七种温度等位基因：

| 等位基因 | 温度区间 | 来源 |
|---------|---------|------|
| 冰冻（freeze） | -2.0 ~ -0.15 | 天然产生 |
| 寒冷（cold） | -0.15 ~ 0.4 | 天然产生 |
| 适中（moderate） | 0.4 ~ 0.85 | 天然产生 |
| 炎热（ardent） | 0.85 ~ 2.0 | 天然产生 |
| 末地（ender） | — | 维度匹配（非天然） |
| 下界（nether） | — | 维度匹配（非天然） |
| 全适应（omni） | — | 任何温度（非天然） |

"天然产生"标记很重要——只有天然等位基因才会在野生蜜蜂身上出现。维度型和全适应型是玩家通过育种才可能获得的"高级"性状。

温度适应性并非孤立判断：蜂箱参数中的温度偏移（由某些设备提供）会影响实际判定值，且温度验证需要湿度的配合。

### 2.4 湿度（Humidity）

类似温度，但判定依据是群系的降水值。内置四种湿度等位基因：

| 等位基因 | 湿度区间 | 来源 |
|---------|---------|------|
| 干燥（dry） | 0 ~ 0.35 | 天然产生 |
| 适中（moderate） | 0.35 ~ 0.65 | 天然产生 |
| 湿润（wet） | 0.65 ~ 1.0 | 天然产生 |
| 全适应（omni） | — | 任何湿度（非天然） |

温湿度共同构成"气候适应"维度。一只"寒冷 + 干燥"的蜜蜂适合在针叶林或雪原工作，而"炎热 + 湿润"的蜜蜂适应丛林。玩家若想让蜜蜂跨群系工作，需通过育种向"全适应"方向推进。

### 2.5 产量（Yield）

产量基因携带一个数值倍率，影响每次采集的产物数量。内置四种：

| 等位基因 | 倍率 |
|---------|------|
| 贫乏（meager） | 0.5× |
| 适中（moderate） | 1.0× |
| 丰富（abundant） | 3.0× |
| 丰收（bumper） | 5.0× |

野生蜜蜂默认产量为贫乏（0.5×），这鼓励玩家通过育种提升产量。

### 2.6 繁殖力（Fertility）

繁殖力基因决定每次繁殖产生的子代数量。内置四种：

| 等位基因 | 子代数 |
|---------|--------|
| 低（poor） | 1 |
| 中（moderate） | 2 |
| 高（high） | 4 |
| 超高（prolific） | 6 |

野生蜜蜂默认为中等繁殖力（2 子代）。高繁殖力能加速种群扩张，但也意味着更多子代需要管理和筛选。

### 2.7 寿命（Lifespan）——"蜜蜂活多久"

寿命基因携带一个倍率，乘以配置中的基准寿命（tick 数）得到实际存活时间。内置四种：

| 等位基因 | 倍率 |
|---------|------|
| 有限（limited） | 0.5× |
| 平均（average） | 1.0× |
| 较长（extended） | 2.0× |
| 极长（exceptional） | 4.0× |

野生蜜蜂默认为平均寿命（1×）。长寿命的蜜蜂性价比更高，是育种的重要目标之一。

## 三、育种机制

### 3.1 二倍体模型

每只蜜蜂的基因组是**二倍体**，即每个基因位点有两份拷贝——一份来自母方，一份来自父方。但在实际性状表现上，系统采用**第一个拷贝（母方来源）作为表现型**（Phenotype），即蜜蜂的实际行为只看第一个基因组的等位基因值。

第二个拷贝虽然不直接表现，但在繁殖时会参与遗传，因此隐性性状可以在后代中重新显现。

### 3.2 杂交

当两只蜜蜂繁殖时，系统通过模拟**染色体交叉互换**来生成子代：

1. 遍历所有基因位点；
2. 对每个位点，随机决定子代从哪个亲代继承该位点的等位基因；
3. 母方和父方拷贝之间还可能发生互换。

这意味着子代的性状可能在双亲之间随机组合。举个例子：

> 亲代 A：温度 = 寒冷，湿度 = 干燥
> 亲代 B：温度 = 炎热，湿度 = 湿润
>
> 子代可能出现：寒冷 + 湿润、炎热 + 干燥、寒冷 + 干燥、炎热 + 湿润中的任意组合。

此外，子代的表现型和隐性拷贝之间也会发生互换，这意味着即使一个性状在亲代表现型中不显示，它仍可能在子代的**表现型**中出现——这是"隔代遗传"效应。

### 3.3 育种策略

玩家培育目标蜜蜂品系通常遵循以下路径：

1. **发现**：在野外找到具有目标性状的蜜蜂（如特定温度适应、特定基础产品）；
2. **杂交组合**：将不同性状的蜜蜂反复繁殖，利用随机重组获得具有期望性状组合的子代；
3. **纯化**：通过同品系自交和多代筛选，使目标性状在隐性拷贝中也固定下来（提高遗传稳定性）；
4. **突变推进**：利用生境和概率触发突变，获得非天然等位基因（如全适应温度）或合成/熔炼产生的新产品；
5. **量产优化**：最终获得高产、高繁殖力、长寿命、全适应且产出特定物品的稳定品系。

## 四、突变系统

突变是蜜蜂获得**非天然等位基因**和**新产物**的唯一途径。突变发生在繁殖生成子代时，时机在杂交完成之后、构建最终基因组之前。

### 4.1 触发机制

突变并非必定发生。每次繁殖中：

1. 系统遍历所有已注册的突变类型，筛除被蜂箱禁用或条件不满足的突变；
2. 累加所有适用突变的基础概率（每种 5%）；
3. 将其与配置中的全局突变概率取较小值，再应用蜂箱参数中的概率倍率；
4. 按各突变在总权重中的占比进行加权随机选择；
5. **最多触发一种突变**（被选中的突变执行后可选择跳过后续，所有内置突变均选择跳过）。

### 4.2 四种突变

**序列追加（append）**
- 触发条件：双亲生境相同。
- 效果：将母方和父方的产品序列交叉合并后分别写入子代双方。例如，母方有 [蜂蜜, 蜂蜡]、父方有 [蜜脾]，子代可能获得 [蜂蜜, 蜂蜡, 蜜脾]。
- 上限：单条序列最多 9 项，超出时从头部截断。

**序列削减（remove）**
- 触发条件：双亲生境相同，且至少一方产品序列长度 > 1。
- 效果：随机移除母方、父方或双方的最后一个产品项。相当于"退化"。
- 设计意图：防止产品序列无限膨胀，维持动态平衡。

**合成突变（crafting）**
- 触发条件：至少一方具有"合成"生境。
- 效果：将产品序列中的物品在工作台配方系统中匹配。如果存在可行的合成配方，整个产品序列被**替换**为合成结果的单个物品。
- 典型应用：产品序列为 [铁锭, 铁锭, 铁锭] → 合成突变 → [铁块]。玩家由此可获得原本无法直接采集的合成物品。

**熔炼突变（smelting）**
- 触发条件：至少一方具有"熔炼"生境。
- 效果：取产品序列**最后一个**物品查询熔炉配方，若能冶炼则替换为冶炼结果（如粗铁 → 铁锭）。
- 与合成突变不同，熔炼突变只替换末尾一项而非整个序列。

### 4.3 突变与生境的互动

合成突变和熔炼突变的触发前提是蜜蜂具有对应生境。这意味着：

- 玩家需要先在蜂箱周围放置对应的生境方块（工作台、熔炉等），让蜜蜂获得该生境；
- 生境本身也是一种可遗传的基因，可通过育种改变；
- 如果玩家不希望突变发生（想保留原始产品序列），只需移除对应生境方块或选择非合成/非熔炼生境的蜜蜂。

## 五、环境适应性

### 5.1 温湿度判定

蜜蜂能否在某个蜂箱工作，取决于其温度和湿度等位基因是否适应当前群系环境。

判定流程：
1. 获取蜂箱所在群系的**基础温度**和**基础湿度**（降水量）；
2. 减去蜂箱参数中的温度/湿度偏移（由附加设备如加热器、加湿器提供）；
3. 用调整后的值匹配蜜蜂的温度和湿度等位基因要求。

注意：温度和湿度是**联合判定**的。温度等位基因的 `isValidFor` 方法接收湿度参数——即使温度匹配，湿度不匹配也会判定为不适应。

### 5.2 生境扫描

在蜂箱周围指定半径（可配置）内，系统扫描每一个方块：
- 检测是否存在任意花朵方块（用于判断"是否有生境"）；
- 检测每个方块是否属于蜜蜂生境关联的方块标签；
- 汇总所有检测到的生境。

如果周围既无花朵也无生境方块，扫描返回空——这意味着蜜蜂在荒漠或地下洞穴中无法正常工作。

### 5.3 产品分配

当蜜蜂的环境验证通过后，系统进行产品产出：

1. 从蜜蜂的表现型基因组中取出产品列表；
2. 按当前蜂箱的**实际生境**过滤产品列表：只有产品项的关联生境与当前生境匹配时才参与产出；
3. 将产出数量按产量基因的倍率计算；
4. 随机分配给过滤后的各产品项（均匀随机），或随机选取一项独占全部数量。

这意味着：**蜜蜂能产出什么，既取决于它的产品基因，也取决于蜂箱周围的生境方块**。同一只蜜蜂在野外（野生生境）和在工作台旁（合成生境）可能产出完全不同的物品。

### 5.4 环境验证事件

系统在判定环境有效性时发出 NeoForge 事件 `BeeEnvironmentValidateEvent`，其他模组可以监听并取消判定。这为模组间联动提供了入口——例如，某个模组可以要求蜜蜂必须在特定结构附近才能工作。

## 六、设计哲学

### 6.1 渐进复杂度

基因组系统的学习曲线设计为**阶梯式**：

- **新手层**：只需理解"不同群系有不同蜜蜂"、"放花蜜蜂才工作"、"繁殖出随机后代"；
- **进阶层**：开始关注温湿度适配、生境类型、产量和繁殖力等数值基因；
- **专家层**：利用合成/熔炼突变定向获取特定物品，通过多代计划育种稳定所有性状。

每一层的机制自然引出下一层的需求，而非一次性灌输所有概念。

### 6.2 可配置性

所有基因位点、等位基因值、突变类型都通过注册表管理。模组包作者可以：
- 添加新的基因位点（在现有七个之外）；
- 添加新的等位基因值（如新的温度区间、新的生境类型）；
- 添加新的突变类型（定义全新的遗传变化逻辑）；
- 通过方块标签控制生境对应哪些方块；
- 通过配置文件调整全局突变概率、基准寿命等参数。

### 6.3 可读性优先

每个基因位点和每个等位基因值都有完整的**翻译键**，支持本地化。UI 展示时使用全名或简写两种模式。产品序列中空列表显示为专用文本（"无产品"），有产品时显示首个产品的物品名称。

### 6.4 数据驱动

基因组的所有数据（等位基因值、产品列表、配方引用）都通过 Minecraft 的 Codec 体系序列化。这意味着：
- 蜜蜂基因可以存储在物品的 NBT/DataComponent 中；
- 配方文件和配置可以直接引用等位基因 ID；
- 网络同步使用高效的整数 ID 而非字符串。

### 6.5 权衡与制约

系统通过多种机制防止玩家"无限优化"：

- **产品序列上限**（9 项）防止序列追加突变无限膨胀；
- **序列削减突变**作为反向力，维持产品序列的动态平衡；
- **非天然标记**标记全适应等高级性状，使其无法在野外自然获取；
- **突变最多触发一种**防止多突变叠加导致不可控结果；
- **生境过滤产品**使同一只蜜蜂在不同环境下有不同表现，避免"全能蜜蜂"；
- **温湿度联合判定**让全适应的获取成为有意义的长期目标。

## 七、典型玩法流程

### 7.1 起步：野外采集

玩家在野外找到自然生成的蜜蜂（根据群系温湿度匹配对应温度/湿度的野生蜜蜂）。初始基因型通常是：

> 温度 = 适中，湿度 = 适中，生境 = 野生，产量 = 贫乏，繁殖力 = 中，寿命 = 平均，产品 = [某种基础产物]

使用捕蜂工具获取蜜蜂并放入蜂箱。在蜂箱周围放置对应生境的花卉，蜜蜂开始工作并产出基础物品。

### 7.2 扩张：杂交育种

将不同群系采集的蜜蜂配对繁殖。利用随机重组获得新的性状组合，例如：

- 炎热群系的蜜蜂 × 湿润群系的蜜蜂 → 可能获得"炎热 + 湿润"的子代，适应丛林；
- 不同产品序列的蜜蜂杂交 + 序列追加突变 → 子代获得更长的产品序列。

同时开始关注产量和繁殖力基因，选择优良子代作为下一轮亲本。

### 7.3 突破：突变驱动

在蜂箱周围放置工作台方块，使蜜蜂获得"合成"生境。将具有合成生境的蜜蜂与具有较长产品序列的蜜蜂杂交，触发合成突变，产品序列被替换为合成结果——由此蜜蜂开始产出原本只能通过工作台合成的物品。

同理，放置熔炉触发熔炼突变，将粗矿冶炼为锭。

### 7.4 精炼：全适应化

通过多代育种和突变，逐步获取"全适应"温度和湿度等位基因。全适应蜜蜂可以在任何群系工作，极大降低蜂箱选址限制。

同时将产量提升至丰收（5×）、繁殖力提升至超高（6 子代）、寿命提升至极长（4×），建立高效自动化生产体系。

### 7.5 终极：定制品系

针对特定目标物品（如某种稀有资源），通过合成突变产生的配方引用，反向追踪并稳定蜜蜂的产品基因。最终获得一只全适应、高产、高繁殖力、长寿命、且稳定产出目标物品的蜜蜂品系——这就是 Beecrasy 基因系统的终极玩法目标。

## 八、鉴定机制与信息博弈

### 8.1 基因的"可见性"

蜜蜂的基因组在刚获得时并非完全可见。`GenomeComponent` 中有一个 `inspected` 标记：

- **未鉴定**（inspected = false）：玩家只能看到蜜蜂的基础外观（如物品名称），无法查看其具体的基因型。在网络同步时，未鉴定基因组发送的是空包，客户端无法渲染基因详情。
- **已鉴定**（inspected = true）：所有基因位点的等位基因值完全公开，可在 UI 中完整展示。

这种设计创造了一个**信息获取**的玩法维度。玩家不是拿到蜜蜂就知晓一切——你需要投入资源进行"鉴定"。

### 8.2 鉴定的游戏意义

鉴定动作由 `GenomeDataHelper.setGenomeInspected(stack)` 执行，通常在玩家使用某种鉴定工具（如放大镜、分析仪）时触发。

鉴定创造的游戏循环：

1. **采集阶段**：玩家在野外四处捕蜂，获得大量未鉴定蜜蜂。此时只能通过蜜蜂产出的物品和蜜蜂所在群系推测其基因型。
2. **筛选阶段**：将采集到的蜜蜂放入蜂箱试运行，观察它们的实际表现（是否适应当前环境、产出什么物品、繁殖出什么后代），淘汰表现不佳的个体。
3. **鉴定阶段**：对表现优异的蜜蜂使用鉴定工具，揭开其完整基因型。此时可以精确制定育种计划。
4. **育种阶段**：基于已知基因型进行有目的的杂交组合。

未鉴定阶段的"信息不透明"迫使玩家依赖**表型观察**——这与真实育种学中的"后裔测定"（progeny testing）原理相通。玩家需要通过蜜蜂的实际行为反推其基因构成，而非直接查看数值。

## 九、蜂箱环境参数

### 9.1 参数聚合

`BeeHiveParameterSet` 是蜂箱工作时的环境快照，它聚合了多重信息：

| 参数来源 | 说明 |
|---------|------|
| 世界与坐标 | 蜂箱所在的具体位置 |
| 群系（Biome） | 决定基础温度、湿度（降水量） |
| 维度类型（DimensionType） | 用于维度型温度等位基因匹配 |
| 方块状态（BlockState） | 蜂箱自身方块的当前状态 |
| 活跃生境（activeBiotopes） | 周边扫描到的生境集合 |
| 工作参数（params） | 附加设备提供的温度偏移、湿度偏移等 |
| 禁用突变（disabledMutation） | 蜂箱或配置禁用的突变 ID 列表 |

### 9.2 参数合并策略

工作参数不是简单覆盖而是**按类型合并**。每个 `BeehiveParameterType` 有自己的 `mergeTo()` 和 `mergeToDefault()` 方法，允许多个设备同时影响同一参数。例如：

- 一个加热器提供 +0.3 温度偏移；
- 另一个冷却器提供 -0.1 温度偏移；
- 合并后实际温度偏移为 +0.2。

这种合并机制让蜂箱的附加设备形成了**策略深度**——玩家需要组合不同的设备来精确调控环境参数，使蜜蜂能在原本不适应的群系中工作。

### 9.3 生境扫描与更新

`GenomeWorkHelper.findBiotope()` 在蜂箱周围指定半径内扫描方块，将扫描结果注入 `activeBiotopes`。每次蜂箱工作周期都会重新扫描（或缓存一段时间），确保环境变化（玩家放置/移除生境方块）能实时反映。

扫描遍历三维空间中的所有方块，检查每个方块是否属于蜜蜂生境关联的方块标签。同时记录是否存在任意花朵方块——即使没有命中具体生境标签，花朵的存在本身也会影响环境判定结果（扫描返回 null 的条件是既无花朵也无生境方块）。

## 十、相似度评分与蜂箱匹配

### 10.1 评分系统的用途

`GenomeWorkHelper.checkSimilarityPoint()` 生成的相似度评分主要用于**蜂箱与蜜蜂的匹配**。当玩家在蜂箱中放入蜜蜂时，系统需要判断这只蜜蜂是否"适合"当前蜂箱的需求。

二倍体评分将母方和父方评分分别打包到 64 位整数的高低 32 位，这使得单次比较就可以涵盖两个等位基因拷贝。

### 10.2 评分权重的设计逻辑

评分权重体现了各基因位点的**重要性梯度**：

| 匹配项 | 分值 | 设计理由 |
|--------|------|---------|
| 产品完全一致 | 100,000 | 产品是蜜蜂的核心价值，完全匹配意味着蜜蜂产出恰好满足蜂箱需求 |
| 每个共同产品 | 10,000 | 部分匹配也有一定价值，允许容忍产品序列的差异 |
| 生境相同 | 1,000 | 生境决定蜜蜂能否产出，重要性仅次于产品 |
| 温度相同 | 100 | 温度和湿度共同构成气候适应，优先级中等 |
| 湿度相同 | 100 | 同上 |
| 其他基因匹配 | 1 | 繁殖力、寿命、产量等数值基因虽然影响效率，但不影响"能否工作" |

这种阶梯式权重意味着：**产品 + 生境匹配是硬门槛**，温湿度是软门槛，数值基因几乎不影响匹配判定。这个设计让玩家在育种时有明确的优先级——先把产品和生境搞对，再优化数值。

## 十一、产品蜜脾与物品系统

### 11.1 蜜脾生成

蜜蜂产出物被封装为"产品蜜脾"（Product Comb）。`GenomeWorkHelper.ProductWithCount` 的 `createProductComb()` 方法生成实际的物品栈：

1. 创建蜜脾基础物品，数量为分配的数量；
2. 将产品物品模板写入 `COMB_PRODUCT` 组件（标识"这个蜜脾包含什么产物"）；
3. 将同样的物品模板写入 `TINT_STACK` 组件（驱动蜜脾的染色渲染）。

这意味着**同一物品 ID 的蜜脾，因内含产品不同，会显示不同的颜色和纹理**。玩家可以通过观察蜜脾的外观判断其内含物，无需打开 UI。

### 11.2 基因组的物品承载

蜜蜂物品通过 `GENOME` 数据组件承载其基因组。`GenomeDataHelper` 的 `setHaploidGenome()` 和 `setDiploidGenome()` 方法在写入基因组时，还会同步写入首个产品物品模板到 `TINT_STACK`，使蜜蜂物品本身也根据其产品基因渲染不同外观。

`GenomeComponent` 支持单倍体（一个 `Genome`）和二倍体（两个 `Genome`）两种模式：
- 野生蜜蜂或未繁殖过的蜜蜂通常是单倍体（仅表现型基因组）；
- 繁殖产生的子代是二倍体（母方 + 父方基因组）。

获取二倍体时，`GenomeDataHelper.getAsDiploid()` 会自动处理这两种情况：单倍体被复制为两份相同基因组，只有一个基因组的二倍体将缺失填充为默认基因组。

## 十二、突变系统的深层设计

### 12.1 为什么最多触发一种突变

`MutationRegistry.handleMutation()` 的加权随机选择一旦命中某个突变就停止（break），这是有意为之：

- **可控性**：一次繁殖只改变一个方面，玩家可以逐一追踪性状变化；
- **计算效率**：突变可能涉及配方查询（合成突变需要遍历工作台配方，熔炼突变需要查询熔炉配方），这些操作开销较大；
- **防滥用**：如果多种突变叠加，可能导致产品序列在单次繁殖中剧烈变化，玩家难以定向选育。

### 12.2 序列追加与削减的生态平衡

序列追加突变和序列削减突变形成了一对**动态平衡力**：

- 追加突变推动产品序列增长，使蜜蜂能产出更多种类的物品；
- 削减突变反向修剪产品序列，防止序列无限膨胀。

两种突变的触发条件完全相同（双亲生境相同），所以它们在同一繁殖中有相同的权重。从长期统计看，序列长度会围绕一个均衡点波动，这个均衡点取决于两种突变的具体概率（均为 5%）和序列长度的临界值（削减突变需要长度 > 1 才触发）。

9 项上限则是硬性天花板，确保序列追加在极端情况下也不会溢出。

### 12.3 合成/熔炼突变的"配方引用"

合成突变和熔炼突变在执行时，会将使用的配方 ID 记录到 `ProductItem.recipe` 中：

```
ProductItem(biotope=CRAFT, recipe=Optional.of("minecraft:iron_block"), stack=...)
```

这个可选的配方引用有几个用途：
- **溯源**：可以追溯蜜蜂产品的"进化路径"——这个铁块是通过 3×3 铁锭合成的，而非直接采集的；
- **再合成**：如果产品序列再次命中同一配方，合成突变可能进一步合成（例如铁块 → 铁砧）；
- **配方兼容性**：模组包作者可以通过检查配方引用了解蜜蜂的产品来源，便于调试和平衡。

### 12.4 合成突变的产品序列匹配

合成突变的配方匹配逻辑分为两个阶段：

1. **精确顺序匹配**：将产品序列中的物品按顺序与配方输入匹配。如果产品的顺序恰好符合某个配方的输入顺序，优先使用该配方。
2. **无序匹配**：如果没有精确顺序匹配，退而使用所有可能的配方（忽略输入顺序）。

这鼓励玩家**按配方顺序排列产品序列**以获得更可预测的合成结果。

## 十三、模组包定制指南

### 13.1 添加新的等位基因值

通过调用 `EnumAlleleType.registerAllele()` 注册自定义等位基因。例如，添加一个新的温度区间：

```java
Temperature.CUSTOM_TEMPERATURE = 
    Alleles.TEMPERATURE.registerAllele(
        new Temperature.RangedTemperature("custom", 0.5f, 1.2f)
    );
```

新等位基因会自动获得翻译键（`allele.<namespace>.<type>.custom`）和整数 ID，无缝融入序列化系统。

### 13.2 添加新的基因位点

通过 `GeneRegistry.register()` 注册新基因。需要提供：
- 唯一 ID
- 等位基因值的 Codec 和 StreamCodec
- 名称转换函数
- 默认值供应者
- 显示优先级

优先级决定了 UI 中的显示顺序。建议新基因的优先级设置在 1000 以上以避免与内置基因冲突（内置基因占用 100~700）。

### 13.3 添加新的突变类型

实现 `Mutation` 接口并通过 `MutationRegistry.register()` 注册。自定义突变可以实现任意复杂的逻辑，例如：

- **环境诱发突变**：仅在特定群系或维度触发的变异；
- **条件产物突变**：根据蜂箱参数添加特殊产物；
- **退化突变**：降低某数值基因的值；
- **跨物种突变**：与其他模组的蜜蜂系统联动。

突变的基础概率（5%）可以在 `getChance()` 中自定义，也可以设计为随蜂箱参数动态变化。

### 13.4 生境方块标签

内置四种生境的方块标签自动生成为 `flowers/biotope_<id>` 格式。模组包作者可以在数据包中添加/修改这些标签，控制哪些方块属于哪个生境。

例如，将某模组的特殊花朵加入 `flowers/biotope_wild` 标签，或创建自定义生境并关联到新的方块标签。

### 13.5 配置文件参数

全局突变概率（`MUTATION_CHANCE`）和基准寿命（`LIFESPAN`）在配置文件中可调。这些参数直接影响游戏节奏：
- 降低突变概率 → 育种过程更长，获得稀有性状更难；
- 提高基准寿命 → 蜜蜂存活更久，降低管理频率。

### 13.6 环境验证事件

`BeeEnvironmentValidateEvent` 是一个可取消的 NeoForge 事件。其他模组可以监听并取消环境有效性判定。典型用例：

- 要求在特定结构（如蜂巢遗迹）附近才允许蜜蜂工作；
- 在特定天气或时间条件下禁用蜜蜂活动；
- 实现跨维度的环境封锁。

## 十四、设计中的取舍与反思

### 14.1 为什么表现型只取第一个拷贝

二倍体基因组有两个拷贝，但表现型（蜜蜂实际行为）只用第一个（母方来源的）拷贝。第二个拷贝仅在繁殖时参与遗传。

**选择这个设计的理由**：
- 简化环境判定逻辑——系统不需要处理两个拷贝不一致时的"显隐性"问题；
- 避免"双倍产出"等平衡问题——如果两个拷贝都表现，产量基因 5× + 5× 会叠加到不可控；
- 保留隐性遗传的可能性——第二个拷贝虽然不表现，但可能在子代表现型中通过交叉互换"浮出水面"。

### 14.2 为什么突变概率统一为 5%

四种内置突变的基础概率均为 `0.05f`。统一概率简化了概率累加计算，且保证各突变在同一繁殖中有相同的被选权。

但接口设计保留了差异化的可能——`getChance()` 接受完整的 `BeeHiveParameterSet` 和 `DiploidGenome` 参数，这意味着未来或自定义突变可以基于当前环境和基因组动态返回不同概率。

### 14.3 为什么不实现完全的"显隐性"

经典遗传学有显性（Dominant）和隐性（Recessive）等位基因的概念。Beecrasy 选择了更简单的模型：**两个拷贝平等参与重组，但只有一个拷贝表现**。

完全的显隐性模型会带来额外的认知负担——玩家需要理解 Aa 杂合子表现 A 性状但携带 a 隐性基因，这在蜜蜂育种中可能过于复杂。当前的简化模型保留了"隔代遗传"的可能性（通过交叉互换），但无需玩家学习显隐性规则。

### 14.4 可扩展性与复杂度控制

注册表模式使系统高度可扩展，但每个扩展点都是"有成本"的：

- 添加新基因位点会使所有蜜蜂的基因组膨胀，增加存储和网络开销；
- 添加新突变类型会增加每次繁殖的计算量（适用性判断 + 概率查询）；
- 添加新等位基因值会稀释已有等位基因的权重（在随机育种中更难命中特定值）。

系统默认设置了七个基因位点和四种突变，这是经过权衡后认为"足够有趣但不至于繁杂"的平衡点。模组包作者在扩展时应考虑这些隐性成本。

---

## 十五、总结

Beecrasy 的基因组系统是一项**用注册表驱动的可扩展遗传模拟**。它借用了真实遗传学的基本框架（基因位点、等位基因、二倍体、杂交、突变），但将复杂度控制在适合 Minecraft 游戏节奏的水平。

从玩家视角，核心体验路径是：

> **采集野生蜜蜂 → 观察表型筛选 → 鉴定揭开基因 → 定向杂交组合 → 环境诱变 → 获得理想品系 → 自动化量产**

每个阶段都有明确的玩家驱动力。基因系统的深度不在于"一次性理解所有机制"，而在于**渐进式发现**——新手可以通过基础育种获得更好的蜜蜂，专家可以精确操作突变和合成路径获取特定产物，而模组包作者可以在不修改核心代码的前提下无限扩展这个系统。
